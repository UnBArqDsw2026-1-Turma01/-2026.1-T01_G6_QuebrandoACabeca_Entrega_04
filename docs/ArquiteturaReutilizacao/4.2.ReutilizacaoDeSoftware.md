## 2. Reutilização de _Software_ Praticada

### 2.1 Reutilização de _Software_ de Terceiros: Bibliotecas e _Frameworks_

Dois dos principais conceitos empregados em reutilização de código, e que foram ambos empregados neste projeto, são as **bibliotecas** e os **_frameworks_**. Ambos caracterizam-se como seções de código reutilizáveis e geralmente prontas criadas para acelerar o desenvolvimento de _software_ resolvendo problemas e capturando funcionalidades (domínios cognitivos) comuns a várias aplicações.

A principal diferença está no **fluxo de controle**: tirando exceções conceitualmente pontuais, na biblioteca, o código do consumidor geralmente chama as funções dela, e a sua base não dita necessariamente a arquitetura global do programa. No _framework_, é ele que controla o fluxo da aplicação e chama o seu código, o chamado Princípio de Hollywood ou Inversão de Controle (IoC, na sigla em inglês), e também define a arquitetura fixa e o ciclo de vida principal.

#### 2.1.1 _Backend_

- **FastAPI**: Reutilização de _framework_ (caixa-cinza) servidor de processos, abstraindo protocolos de rede puros;
- **SQLAlchemy**: Reutilização de servidor de gerenciamento transacional de banco de dados, poupando a equipe de lidar com sintaxe SQL bruta, assim tanto reduzindo a quantidade de código novo necessária como melhorando a segurança contra eventuais falhas como injeção de SQL.

#### 2.1.2 _Frontend_

- **React Router DOM**: Reaproveitamento de mecânica de navegação SPA (_Single Page Application_) sem necessidade de recarregar a visualização da página por inteiro, permitindo ao usuário navegação mais rápida e atualizações de conteúdo dinâmicas;
- Tipagens do **TypeScript**: Reaproveitamento de interfaces e definições de tipos de dados (como as modelagens em `game.types.ts` e `user.types.ts`) por múltiplos componentes, _hooks_ e chamadas de API, estabelecendo contratos únicos que evitam a reescrita de validações de estrutura de dados ao longo do código-fonte.

### 2.2 Reutilização de Lógica via Padrões de Projeto (_Backend_)

**Padrões de projeto** são elementos arquiteturais menores, mais abstratos e menos especializados do que _frameworks_. Não se manifestam como código por si só, sendo efetivamente uma assistência ou solução puramente conceitual e abstrata que habilita a criação de código de um _framework_.

No _backend_, foram aplicados extensamente padrões de projeto (_design patterns_) do GoF (`backend/app/patterns/`) que viabilizam a reutilização de soluções genéricas para problemas recorrentes.

#### Exemplo 1: Padrão _Factory_ para criação de Dificuldades

- **Conceito:** O padrão _Factory_ permite encapsular a lógica de criação de objetos, garantindo que toda regra de fabricação fique em uma fonte unificada, evitando a cópia destas lógicas pelos _routers_.
- **Código Comprobatório:** _Factory_ da classe abstrata de **dificuldade** de jogo (`Difficulty`) no _backend_, que pode ser Fácil, Média ou Difícil (`Easy`/`Medium`/`Hard`):

```python
def get_difficulty(dto: GetToyDifficultyPayload) -> ResultPayload[Difficulty]:
    """
    Importa e instancia a classe de dificuldade selecionada dinamicamente.
    Args:
        selected_difficulty (str): A dificuldade selecionada.
    Returns:
        Difficulty: A instância da dificuldade selecionada.
    """
    selected_difficulty = dto.difficulty

    try:
        # Reuso de lógica: em vez de múltiplos blocos "if/else" para cada dificuldade,
        # importa-se dinamicamente a respectiva classe, permitindo escalar o jogo para
        # novas dificuldades no futuro sem alterar este contrato da Factory.
        module = importlib.import_module(
            f".{selected_difficulty.lower()}", package=__package__
        )

        class_name = f"{selected_difficulty.capitalize()}Difficulty"
        backend_class = getattr(module, class_name)

        if not issubclass(backend_class, Difficulty):
            raise ValueError(
                f"Class '{class_name}' in '{selected_difficulty}.py' module does not implement Difficulty."
            )

        return ResultPayload(
            success=True, message=["Was able to get difficulty!"], data=backend_class()
        )
    except ImportError:
        return ResultPayload(
            success=False,
            error=[f"Difficulty '{selected_difficulty}' not found."],
        )

    except AttributeError:
        return ResultPayload(
            success=False,
            error=[
                f"Class '{class_name}' not found in '{selected_difficulty}.py' module."
            ],
        )
```

#### Exemplo 2: Padrão _Builder_ para construção do Quebra-Cabeça

- **Conceito:** O padrão _Builder_ gerencia a construção de instâncias complexas (como a geração do tabuleiro respeitando diferentes dificuldades: fácil, médio, difícil), encapsulando a ordem e os detalhes da criação.
- **Código Comprobatório:** O `Director` orquestra a construção reutilizando a mesma sequência de passos para diferentes tipos de quebra-cabeça:

```python
# backend/app/patterns/builder/director.py
class Director:
    @staticmethod
    def criar_quebra_cabeca(construtor: PuzzleConstrutor) -> ResultPayload:
        """Orquestra a construção do quebra-cabeça, reutilizando a mesma lógica para diferentes dificuldades."""
        construtor.construir_grade()
        construtor.construir_pecas()
        construtor.aplicar_efeito()
        return construtor.obter_resultado()
```

### 2.3 Componentes base (_Frontend_: `components/common/`)

No desenho de um sistema de _software_, um **componente** é um elemento modular que encapsula funcionalidades específicas atrás de um conjunto de interfaces externas, proporcionando coesão, reuso e robustez no sistema e seu desenvolvimento. Um componente pode ser substituído por outro com as mesmas interfaces e, potencialmente, ter estrutura interna.

Certas bibliotecas e _frameworks_ encorajam a criação e o reuso de componentes de _software_ como um paradigma central de desenvolvimento, a exemplo, dentro do _frontend_ deste próprio projeto, da biblioteca gráfica **React**.

- **Conceito** do Exemplo: Construção de botões, modais e _layouts_ de fundo unificados. A interface herda as propriedades, permitindo chamá-los na página de Login, Jogo ou Configurações sem recriar _layouts_.

- **Código Comprobatório:** Classes `Card` e `Button`, e seu uso na página de Login:

#### `frontend/src/components/common/Card/Card.tsx`:

```typescript
export interface CardProps extends HTMLAttributes<HTMLDivElement> {
  /** Aplica .card-interactive (hover com leve elevação) — usar em cards clicáveis, como seleção de nível/dificuldade. */
  interactive?: boolean;
  children: ReactNode;
}

export function Card({ interactive = false, children, className, ...rest }: CardProps) {
  // Agrupa classes condicionalmente, centralizando a lógica de estilo base da interface
  const classNames = ['card', interactive ? 'card-interactive' : '', className].filter(Boolean).join(' ');

  return (
    // Todas as props dinâmicas (...rest) e o conteúdo (children) são repassados, permitindo reuso universal flexível
    <div className={classNames} {...rest}>
      {children}
    </div>
  );
}
```

#### `frontend/src/components/common/Button/Button.tsx`:

```typescript
export type ButtonVariant = 'primary' | 'secondary' | 'success' | 'danger';

export interface ButtonProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: ButtonVariant;
  /** Desabilita o botão e aplica o mesmo estilo de :disabled já definido em global.css. */
  isLoading?: boolean;
  children: ReactNode;
}

const VARIANT_CLASS: Record<ButtonVariant, string> = {
  primary: 'btn-primary',
  secondary: 'btn-secondary',
  success: 'btn-success',
  danger: 'btn-danger',
};

export function Button({
  variant = 'primary',
  isLoading = false,
  disabled,
  children,
  className,
  ...rest
}: ButtonProps) {
  // O dicionário VARIANT_CLASS garante que a aplicação inteira utilize o mesmo dicionário de botões tipados
  const classNames = ['btn', VARIANT_CLASS[variant], className].filter(Boolean).join(' ');

  return (
    // Estados repetitivos como 'isLoading' são gerenciados e renderizados internamente apenas uma vez
    <button className={classNames} disabled={disabled || isLoading} aria-busy={isLoading} {...rest}>
      {children}
    </button>
  );
}
```

### 2.4 Serviços (_Frontend_ & _Backend_)

A utilização de serviços promove a reutilização, a coesão e o desacoplamento ao centralizar lógicas de integração e exposição de regras de negócio em módulos independentes da interface. No **Quebrando a Cabeça**, adotamos a separação por serviços com responsabilidades bem definidas em ambos os subsistemas.

#### 2.4.1 Serviços de Comunicação (_Frontend_)

No _frontend_, as requisições à API REST do _backend_ são totalmente abstraídas em arquivos de serviços dedicados dentro do diretório `frontend/src/api/` (tais como `authApi.ts`, `puzzleApi.ts` e `scoreApi.ts`). Isso evita a repetição de configuração de clientes HTTP (como instâncias de _Axios_ ou _Fetch API_) e cabeçalhos de autenticação dentro da renderização dos componentes Visuais, além de disponibilizar clientes prontos para uso em qualquer página e contexto (seja no `Jogo` ou no `Menu`).

- **Código Comprobatório:** Exemplo hipotético de estrutura centralizando lógicas de placar em `scoreApi.ts` e a chamada `httpClient` base.

##### `frontend/src/api/scoreApi.ts`:

```typescript
import { httpClient } from "./httpClient";

export const scoreApi = {
  getLeaderboard: async () => {
    // Reutiliza o httpClient pré-configurado
    const response = await httpClient.get("/scores/leaderboard");
    return response.data;
  },
  submitScore: async (data: ScorePayload) => {
    const response = await httpClient.post("/scores", data);
    return response.data;
  },
};
```

#### 2.4.2 _Routers_ de _Endpoints_ REST (_Backend_)

No _backend_ em **FastAPI**, utilizamos módulos controladores encapsulados em _Routers_ (`auth_router.py`, `puzzle_router.py`, `score_router.py`, etc, contidos em `backend/app/api/routers/`). Cada módulo agrupa sub-rotas e se expõe como um serviço provendo capacidades para manipulação da lógica do quebra-cabeça, gestão de contas e placares. O uso desses controladores como serviço no _backend_ viabiliza que qualquer cliente externo (a atual página _Web_ ou num futuro um aplicativo _Mobile_) possa fazer uso e reutilizar toda a operação de processamento sem retrabalhar em lógicas.

- **Código Comprobatório:** Em `main.py`, a importação e o reuso de rotas independentes montam o ecossistema da aplicação facilmente:

##### `backend/app/main.py`:

```python
from fastapi import FastAPI
from app.api.routers import auth_router, puzzle_router, score_router, session_router

app = FastAPI(title="Quebrando a Cabeça API")

# O framework reutiliza as as sub-redes pré-configuradas nos routers
app.include_router(auth_router.router, prefix="/auth", tags=["Auth"])
app.include_router(puzzle_router.router, prefix="/puzzle", tags=["Puzzle"])
app.include_router(score_router.router, prefix="/score", tags=["Scores"])
```

## 3. Comentários Gerais sobre o Trabalho em Equipe

A equipe trabalhou de forma colaborativa para identificar oportunidades de reuso desde o início do desenvolvimento. No _frontend_, definimos um conjunto de componentes comuns (`Button`, `Card`, `Popup`) e _hooks_ customizados (`useImageUpload`, `useGame`) que foram utilizados em múltiplas telas, reduzindo a duplicação de código e garantindo consistência visual.
