## 1. Introdução

### 1.1 Objetivo

Este documento tem como objetivo apresentar uma visão abrangente da arquitetura do sistema **Quebrando a Cabeça**, por meio de múltiplas visões arquiteturais. Ele serve como guia de referência para desenvolvedores, arquitetos e demais stakeholders, descrevendo as decisões arquiteturais tomadas, os padrões adotados e a organização estrutural do software.

O documento segue o modelo de **Visão 4+1**, proposto por Philippe Kruchten (1995) e adotado pelo RUP (Rational Unified Process), apresentando obrigatoriamente a **Visão Lógica** e, como segunda visão escolhida pela equipe, a **Visão de Implementação**.

### 1.2 Escopo

O sistema **Quebrando a Cabeça** é uma aplicação web que tem por finalidade permitir que usuários montem quebra-cabeças digitais a partir de imagens próprias ou predefinidas, com diferentes níveis de dificuldade e configurações de partida.

Este DAS abrange a estrutura arquitetural completa do sistema, organizada em dois subsistemas interdependentes:

#### Subsistema Frontend

Desenvolvido em **React com TypeScript**, responsável pela interface do usuário, navegação entre telas, upload de imagem, seleção de dificuldade e renderização do tabuleiro do quebra-cabeça. Organizado em módulos de páginas, componentes reutilizáveis, hooks customizados e serviços de comunicação.

#### Subsistema Backend

Desenvolvido em **Python com FastAPI**, responsável pelo processamento da lógica de negócio do jogo, incluindo a construção e validação do quebra-cabeça, seleção e nomeação de efeitos visuais (cuja renderização é executada pelo frontend) e gerenciamento de sessões multiplayer e pontuações.

#### Integração entre Subsistemas

A comunicação entre os subsistemas ocorre por meio de uma **API REST**: o frontend envia a imagem selecionada e a dificuldade configurada pelo usuário, e o backend retorna as peças processadas e o estado do tabuleiro. Essa integração é encapsulada no frontend por **módulos de serviço dedicados no diretório `api/` (tais como `puzzleApi.ts`, `authApi.ts` e `scoreApi.ts`)**, isolando os componentes visuais dos detalhes de comunicação com o servidor e chamadas HTTP.

Além da geração do quebra-cabeça, a API REST também fornece endpoints para **autenticação** (login, cadastro, recuperação de senha) e para **registro de pontuações e ranking**, garantindo a persistência dos dados de progresso do usuário.

## 2. Representação Arquitetural

O sistema **Quebrando a Cabeça** é construído como uma **Single Page Application (SPA)** no frontend e uma **API REST** no backend, comunicando-se por meio de requisições HTTP. A arquitetura adotada segue princípios de **componentização**, **separação de responsabilidades** e **reutilização**, organizando o código em camadas de apresentação, lógica de negócio e serviços.

### Subsistema Frontend

Desenvolvido com **React + TypeScript**, organiza o código em camadas de apresentação, lógica de negócio local e serviços de comunicação com o backend.

| Tecnologia          | Papel Arquitetural                                                |
| ------------------- | ----------------------------------------------------------------- |
| React 18+           | Biblioteca principal para construção de UI baseada em componentes |
| TypeScript          | Tipagem estática, aumentando robustez e manutenibilidade          |
| React Router DOM    | Gerenciamento de rotas e navegação entre telas (SPA)              |
| CSS Modules / CSS   | Estilos encapsulados por componente, evitando conflitos globais   |
| Context API / Hooks | Gerenciamento de estado local e compartilhado entre componentes   |

### Subsistema Backend

Desenvolvido com **Python + FastAPI**, expõe os serviços de negócio do jogo por meio de endpoints REST consumidos pelo frontend.

| Tecnologia | Papel Arquitetural                                                                  |
| ---------- | ----------------------------------------------------------------------------------- |
| Python 3.x | Linguagem principal do backend, responsável pela lógica de negócio do jogo          |
| FastAPI    | Framework web para exposição da lógica de negócio via endpoints REST                |
| Uvicorn    | Servidor ASGI responsável pela execução da aplicação FastAPI                        |
| Pydantic   | Validação e serialização dos payloads de entrada e saída (DTOs)                     |
| SQLAlchemy | ORM para mapeamento objeto-relacional e acesso ao banco de dados                    |
| SQLite     | Banco de dados relacional local — arquivo `quebra_cabeca.db` gerado automaticamente |

### Infraestrutura

| Tecnologia     | Papel Arquitetural                                                                            |
| -------------- | --------------------------------------------------------------------------------------------- |
| Docker         | Containerização dos subsistemas frontend e backend                                            |
| Docker Compose | Orquestração dos containers, facilitando a execução do ambiente completo com um único comando |

### Integração entre Subsistemas

A comunicação entre frontend e backend ocorre via **API REST sobre HTTP**. O frontend encapsula toda a comunicação no módulo `api/puzzleApi.ts`, enviando a imagem selecionada (em base64) e a dificuldade configurada, e recebendo as peças processadas e o estado do tabuleiro.

| Camada de Integração | Papel Arquitetural                                                   |
| -------------------- | -------------------------------------------------------------------- |
| `api/puzzleApi.ts`   | Módulo frontend que encapsula as chamadas HTTP à API REST do backend |
| API REST (FastAPI)   | Interface de contrato entre os dois subsistemas                      |
| CORS Middleware      | Permite a comunicação entre origens distintas (frontend e backend)   |

## 3. Metas e Restrições Arquiteturais

### 3.1 Metas

| ID  | Meta                           | Descrição                                                                                              |
| --- | ------------------------------ | ------------------------------------------------------------------------------------------------------ |
| M01 | Manutenibilidade               | O código deve ser organizado em componentes coesos, facilmente alteráveis sem impacto em outras partes |
| M02 | Reusabilidade                  | Componentes de UI (ex.: botões, cards, headers) devem ser genéricos e reutilizáveis entre telas        |
| M03 | Separação de responsabilidades | Lógica de negócio, estado e apresentação devem estar claramente separados                              |
| M04 | Escalabilidade                 | Novas telas e funcionalidades devem ser adicionáveis sem refatoração estrutural                        |
| M05 | Usabilidade                    | A navegação deve ser intuitiva e a interface responsiva para diferentes resoluções                     |

| ID  | Restrição                                | Descrição                                                                            |
| --- | ---------------------------------------- | ------------------------------------------------------------------------------------ |
| R01 | Tecnologia front-end                     | A aplicação deve ser implementada exclusivamente com React e TypeScript              |
| R02 | Navegadores alvo                         | Compatibilidade com Chrome, Firefox e Edge em suas versões recentes                  |
| R03 | Sem dependência de framework CSS externo | Os estilos devem ser mantidos em arquivos `.css` próprios do projeto                 |

## 5. Visão Lógica

### 5.1 Visão Geral

A Visão Lógica descreve a **organização conceitual do software** em termos de camadas, pacotes, subsistemas, classes e interfaces. Ela mostra como as principais funcionalidades do sistema estão estruturadas do ponto de vista do design, sem abordar aspectos físicos de implantação ou código-fonte.

Conforme definição apresentada pela Profª. Milene Serrano:

> _"A Visão Lógica organiza conceitualmente o software em termos de camadas, subsistemas, pacotes, frameworks, classes, interfaces e realizações de caso de uso."_

### Backend

| Camada             | Localização                 | Responsabilidade                                                                                                       |
| ------------------ | --------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **Aplicação**      | `api/routers/`              | Endpoints REST, validação de entrada (Pydantic), orquestração de domínio e persistência                                |

- **`pages/`** — Contém os componentes de tela completa da aplicação. Cada arquivo `.tsx` corresponde a uma rota navegável: `Login`, `Cadastro`, `Menu`, `SelecaoNivel`, `SelecaoDificuldade`, `UploadImagem`, `Jogo`, `Vitoria`, entre outras. - **`components/`** — Agrupa os componentes reutilizáveis de UI: `Header`, `LevelCard`, `Popup`, `Toggle`, `DifficultyCard`. Seguem o princípio de responsabilidade única e não possuem lógica de negócio.

- **`api/`** — Módulo responsável pela comunicação com o backend. Isola as demais camadas de detalhes de implementação de I/O (`authApi.ts`, `puzzleApi.ts`, `scoreApi.ts`).

- **`utils/`** — Funções utilitárias puras (sem efeitos colaterais): validadores de tipo e tamanho de arquivo, formatadores de dados, constantes globais.

- **`types/`** — Definições TypeScript globais: interfaces, tipos e enums compartilhados entre camadas.

### 5.5 Decisões de Design do Backend

#### Separação entre Domínio e Aplicação

O backend é estruturado em duas camadas conceituais bem definidas, cada uma com responsabilidades específicas:

| Camada        | Localização    | Responsabilidade                                                                         |
| ------------- | -------------- | ---------------------------------------------------------------------------------------- |
| **Aplicação** | `api/routers/` | Exposição de endpoints REST, validação de dados de entrada e persistência via SQLAlchemy |
| **Domínio**   | `domain/game/` | Regras de negócio do jogo, independentes de infraestrutura, HTTP ou frameworks externos  |

A camada de domínio é composta pelas classes `Piece`, `Move` e `PuzzleBoard`, que representam os conceitos centrais do jogo — peça, movimento e tabuleiro — sem qualquer vínculo com banco de dados ou camada de apresentação.

---

#### Desacoplamento Estratégico

Por decisão arquitetural, essas camadas são **intencionalmente desacopladas**, conforme detalhado a seguir:

- **Geração de peças**  
  Os routers utilizam o padrão **Builder** (`ConstrutorFacil`, `ConstrutorMedio`, `ConstrutorDificil`) para gerar peças com coordenadas normalizadas no intervalo `0.0–1.0`, garantindo que os dados estejam prontos para consumo direto pelo frontend.

- **Validação de encaixe**  
  A verificação de compatibilidade entre peças é delegada à classe `PecaUnica`, que implementa o padrão **Composite**. Essa classe centraliza a regra de tolerância (`_TOLERANCIA = 0.02`) no domínio, evitando duplicação de lógica na camada de aplicação.

- **Expressividade do domínio**  
  As classes `PuzzleBoard` e `Move` documentam claramente a intenção de negócio e viabilizam a realização de testes unitários focados exclusivamente nas regras do jogo, sem dependência da API.

---

#### Benefícios Arquiteturais

Essa organização está alinhada ao princípio de **separação de responsabilidades** e confere ao sistema maior flexibilidade evolutiva. Por exemplo:

- Uma futura migração do banco de dados SQLite para PostgreSQL pode ser realizada sem que qualquer classe de domínio seja afetada, uma vez que a lógica de negócio permanece isolada da infraestrutura.
- A manutenção e evolução das regras do jogo tornam-se mais seguras e previsíveis, pois alterações na camada de domínio não interferem nos mecanismos de exposição da API ou persistência de dados.

## 7. Padrões e Estilos Arquiteturais

Esta seção evidencia os **padrões e estilos arquiteturais** aplicados no projeto, com indicação do código que os comprova, conforme exigido na entrega (código comprobatório rodando).

### 7.1 Estilo Arquitetural: Componentes e Conectores (C&C)

O sistema segue o estilo **Component-and-Connector**, onde:

- **Componentes** são os arquivos `.tsx` (LevelSelection, ImageUpload, etc.)
- **Conectores** são as interfaces de comunicação: `props`, `Context API` e `hooks`

Este estilo é amplamente recomendado para aplicações React e favorece reuso e testabilidade independente.

### 7.2 Padrão Arquitetural: Camadas (Layered Architecture)

A organização `pages/ → components/ → hooks/ → api/` implementa uma **arquitetura em camadas**, onde:

- Camadas superiores dependem de camadas inferiores
- Camadas inferiores nunca dependem de camadas superiores (princípio da dependência unidirecional)

```typescript
// Exemplo em LevelSelection.tsx (Camada de Apresentação):
// A camada de página consome a funcionalidade de um hook customizado (camada inferior)
const { preview, processFile, resetUpload } = useImageUpload();
// ↑ páginas (camada alta) consomem hooks (camada mais baixa)
// Desta forma, preserva-se o desacoplamento: o hook não conhece a interface gráfica.
```

### 7.3 Padrão de Design: Context + Reducer (State Management)

O gerenciamento de estado global segue o padrão **Context + Hooks**, substituindo dependências externas, como Redux, para fornecer um estado de aplicação robusto e evitar o _prop-drilling_ (cascata de propriedades passadas manualmente de pai para filho):

```typescript
// GameContext.tsx — padrão Context Provider
export const GameContext = createContext<GameContextType | null>(null);

export const GameProvider: React.FC<{ children: React.ReactNode }> = ({ children }) => {
  const [selectedLevel, setSelectedLevel] = useState<CardData | null>(null);
  const [difficulty, setDifficulty] = useState<DifficultyLevel>("medio");

  return (
    <GameContext.Provider value={{ selectedLevel, setSelectedLevel, difficulty, setDifficulty }}>
      {children}
    </GameContext.Provider>
  );
};
```

### 7.4 Padrão de Design: Custom Hooks (Separação de Lógica)

A lógica de negócio é extraída dos componentes visuais para **hooks customizados**, seguindo o princípio de separação de responsabilidades:

```typescript
// useImageUpload.ts — encapsula toda a lógica de upload
export const useImageUpload = () => {
  const [preview, setPreview] = useState<FilePreview | null>(null);
  const [dragging, setDragging] = useState(false);

  const processFile = useCallback((file: File | null | undefined) => {
    if (!file) return;
    if (!ACCEPTED_TYPES.includes(file.type)) return;
    if (file.size / 1024 / 1024 > MAX_SIZE_MB) return;
    // leitura e atualização de estado...
  }, []);

  return {
    preview,
    dragging,
    processFile,
    resetUpload,
    handleDrop,
    handleDragOver,
    handleDragLeave,
  };
};
```

### 7.5 Padrão: CSS Variables + Design Tokens

O sistema de estilização utiliza **CSS Custom Properties (variáveis)** como design tokens globais, garantindo consistência visual e manutenibilidade do tema:

```css
/* Definição global (compartilhada entre todos os componentes) */
:root {
  --bg: #0f0f14;
  --surface: #1a1a24;

  --accent2: #f0a500;
  --success: #34c98a;
  --font: "Outfit", sans-serif;
  --mono: "Space Mono", monospace;
}
```

### 7.6 Separação Domínio/Aplicação (Backend)

As classes `Piece`, `Move` e `PuzzleBoard` em `domain/game/` formam uma **camada de domínio rico** isolada da camada de aplicação (`api/routers/`), seguindo o princípio de independência de infraestrutura.

A geração do quebra-cabeça, por exemplo, é inteiramente responsabilidade do domínio — o _router_ da aplicação apenas orquestra a chamada, sem precisar conhecer os detalhes de construção, como a matemática de matrizes ou o recorte virtual da imagem:

```python
# puzzle_router.py — Controlador da camada de aplicação

# A Aplicação delega dinamicamente a definição das regras à Factory de Domain (_BUILDERS)
builder_class = _BUILDERS[req.difficulty]

# O Router instrui o Director a construir as peças via abstração, independente da API e sem "poluir" o endpoint REST
result_builder = Director().criar_quebra_cabeca(builder_class())

if not result_builder.success:
    raise HTTPException(status_code=500, detail=result_builder.error)

qc = result_builder.data  # qc: Objeto de domínio puro, pronto sem lidar com HTTP ou infraestrutura
```

Essa separação segue o mesmo princípio aplicado no frontend (seção 7.2), onde camadas superiores dependem de camadas inferiores e nunca o contrário.

### 7.7 Estilo Arquitetural: Cliente-Servidor via API REST

O projeto segue globalmente o estilo arquitetural **Cliente-Servidor**. O _frontend_ (React) atua como cliente consumindo serviços providos pelo _backend_ (FastAPI), que expõe seus recursos por meio de uma **API RESTful**. Esse estilo garante o desacoplamento das evoluções tecnológicas entre a interface e o processamento estrito de regras de negócio.

### 7.8 Padrões Criacionais (Builder e Factory)

No _backend_, a criação de um quebra-cabeça envolve propriedades flexíveis. O padrão **Factory** é utilizado para determinar e instanciar dinamicamente as regras de tabuleiro através da rota escolhida (classe genérica `Difficulty` instanciando `Easy`, `Medium` ou `Hard`). Aliado a ele, o padrão **Builder** recebe os parâmetros da imagem e da fábrica para montar as propriedades finais de contorno e posições matemáticas das peças do tabuleiro, blindando as rotas da complexidade criacional.

### 7.9 Padrões Estruturais e Comportamentais (Composite e Strategy)

O padrão **Composite** foi aplicado para estabelecer as hierarquias no jogo, permitindo modelar a relação árvore-folha de um "Pedaço Único" e um "Grupo de Peças encaixadas" delegando colisões e posições de forma unificada. Adicionalmente, o **Strategy** foi adotado para o mapeamento dinâmico de filtros visuais (como tons de cinza ou linhas estéticas de Jigsaw), isolando o algoritmo gerador de imagem da lógica principal e permitindo escalar opções de estilo futuras.
