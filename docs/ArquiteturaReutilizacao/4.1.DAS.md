## 1. Introdução

### 1.1 Objetivo

Este documento tem como objetivo apresentar uma visão abrangente da arquitetura do sistema **Quebrando a Cabeça**, por meio de múltiplas visões arquiteturais. Ele serve como guia de referência para desenvolvedores, arquitetos e demais stakeholders, descrevendo as decisões arquiteturais tomadas, os padrões adotados e a organização estrutural do software.

O documento segue o modelo de **Visão 4+1**, proposto por Philippe Kruchten (1995) e adotado pelo RUP (Rational Unified Process), apresentando obrigatoriamente a **Visão Lógica** e, como segunda visão escolhida pela equipe, a **Visão de Implementação**.

### 1.2 Escopo

O sistema **Quebrando a Cabeça** é uma aplicação web que tem por finalidade permitir que usuários montem quebra-cabeças digitais a partir de imagens próprias ou predefinidas, com diferentes níveis de dificuldade e configurações de partida.

Este DAS abrange a estrutura arquitetural completa do sistema, organizada em dois subsistemas interdependentes:

#### Subsistema Frontend

Desenvolvido em **React com TypeScript**, responsável pela interface do usuário, navegação entre telas, upload de imagem, seleção de dificuldade e renderização do tabuleiro do quebra-cabeça. Organizado em módulos de páginas, componentes reutilizáveis, hooks customizados e serviços de comunicação.

#### Subsistema Backend

Desenvolvido em **Python com FastAPI**, responsável pelo processamento da lógica de negócio do jogo, incluindo a construção e validação do quebra-cabeça, aplicação de efeitos visuais sobre a imagem e gerenciamento de sessões multiplayer.

#### Integração entre Subsistemas

A comunicação entre os subsistemas ocorre por meio de uma **API REST**: o frontend envia a imagem selecionada e a dificuldade configurada pelo usuário, e o backend retorna as peças processadas e o estado do tabuleiro. Essa integração é encapsulada no frontend pelo módulo `services/puzzleService.ts`, isolando os demais componentes dos detalhes de comunicação com o servidor.

# 2. Representação Arquitetural

O sistema **Quebrando a Cabeça** é construído como uma **Single Page Application (SPA)** no frontend e uma **API REST** no backend, comunicando-se por meio de requisições HTTP. A arquitetura adotada segue princípios de **componentização**, **separação de responsabilidades** e **reutilização**, organizando o código em camadas de apresentação, lógica de negócio e serviços.

## Subsistema Frontend

Desenvolvido com **React + TypeScript**, organiza o código em camadas de apresentação, lógica de negócio local e serviços de comunicação com o backend.

| Tecnologia          | Papel Arquitetural                                                        |
|---------------------|---------------------------------------------------------------------------|
| React 18+           | Biblioteca principal para construção de UI baseada em componentes         |
| TypeScript          | Tipagem estática, aumentando robustez e manutenibilidade                  |
| React Router DOM    | Gerenciamento de rotas e navegação entre telas (SPA)                      |
| CSS Modules / CSS   | Estilos encapsulados por componente, evitando conflitos globais            |
| Context API / Hooks | Gerenciamento de estado local e compartilhado entre componentes           |

## Subsistema Backend

Desenvolvido com **Python + FastAPI**, expõe os serviços de negócio do jogo por meio de endpoints REST consumidos pelo frontend.

| Tecnologia | Papel Arquitetural                                                              |
|------------|---------------------------------------------------------------------------------|
| Python 3.x | Linguagem principal do backend, responsável pela lógica de negócio do jogo      |
| FastAPI    | Framework web para exposição da lógica de negócio via endpoints REST            |
| Uvicorn    | Servidor ASGI responsável pela execução da aplicação FastAPI                    |
| Pydantic   | Validação e serialização dos payloads de entrada e saída (DTOs)                 |

## Integração entre Subsistemas

A comunicação entre frontend e backend ocorre via **API REST sobre HTTP**. O frontend encapsula toda a comunicação no módulo `services/puzzleService.ts`, enviando a imagem selecionada (em base64) e a dificuldade configurada, e recebendo as peças processadas e o estado do tabuleiro.

| Camada de Integração      | Papel Arquitetural                                                        |
|---------------------------|---------------------------------------------------------------------------|
| `services/puzzleService.ts` | Módulo frontend que encapsula as chamadas HTTP à API REST do backend    |
| API REST (FastAPI)        | Interface de contrato entre os dois subsistemas                           |
| CORS Middleware           | Permite a comunicação entre origens distintas (frontend e backend)        |

## 3. Metas e Restrições Arquiteturais

### 3.1 Metas

| ID  | Meta                        | Descrição                                                                                               |
|-----|-----------------------------|---------------------------------------------------------------------------------------------------------|
| M01 | Manutenibilidade            | O código deve ser organizado em componentes coesos, facilmente alteráveis sem impacto em outras partes  |
| M02 | Reusabilidade               | Componentes de UI (ex.: botões, cards, headers) devem ser genéricos e reutilizáveis entre telas         |
| M03 | Separação de responsabilidades | Lógica de negócio, estado e apresentação devem estar claramente separados                           |
| M04 | Escalabilidade              | Novas telas e funcionalidades devem ser adicionáveis sem refatoração estrutural                         |
| M05 | Usabilidade                 | A navegação deve ser intuitiva e a interface responsiva para diferentes resoluções                       |

### 3.2 Restrições

| ID  | Restrição                   | Descrição                                                                               |
|-----|-----------------------------|-----------------------------------------------------------------------------------------|
| R01 | Tecnologia front-end        | A aplicação deve ser implementada exclusivamente com React e TypeScript                 |
| R02 | Navegadores alvo            | Compatibilidade com Chrome, Firefox e Edge em suas versões recentes                     |
| R03 | Sem dependência de framework CSS externo | Os estilos devem ser mantidos em arquivos `.css` próprios do projeto    |
| R04 | Fontes externas             | Apenas Google Fonts é permitido como fonte externa, sem bibliotecas de UI (ex.: MUI)    |

## 5. Visão Lógica

### 5.1 Visão Geral

A Visão Lógica descreve a **organização conceitual do software** em termos de camadas, pacotes, subsistemas, classes e interfaces. Ela mostra como as principais funcionalidades do sistema estão estruturadas do ponto de vista do design, sem abordar aspectos físicos de implantação ou código-fonte.

Conforme definição apresentada pela Profa. Milene Serrano:

> *"A Visão Lógica organiza conceitualmente o software em termos de camadas, subsistemas, pacotes, frameworks, classes, interfaces e realizações de caso de uso."*

No contexto deste projeto, a arquitetura lógica é composta por **três camadas principais**:

| Camada        | Responsabilidade                                                                      |
|---------------|---------------------------------------------------------------------------------------|
| **Apresentação** | Componentes React responsáveis pela renderização da UI e interação com o usuário   |
| **Aplicativo**   | Lógica de navegação, gerenciamento de estado (hooks/context) e regras de negócio  |
| **Serviços**     | Integração com APIs externas, File API do browser, e utilitários transversais      |

**Descrição dos pacotes principais:**

- **`pages/`** — Contém os componentes de tela completa da aplicação. Cada arquivo `.tsx` corresponde a uma rota navegável: `LevelSelection`, `ImageUpload`, `DifficultySelection` e `Game`. São os componentes de nível mais alto na hierarquia de apresentação.

- **`components/`** — Agrupa os componentes reutilizáveis de UI: `Header`, `LevelCard`, `Popup`, `Toggle`, `DifficultyCard`. Seguem o princípio de responsabilidade única e não possuem lógica de negócio.

- **`context/`** — Implementa o gerenciamento de estado global via Context API do React. O `GameContext` centraliza as configurações de partida (nível selecionado, dificuldade, imagem) que precisam ser compartilhadas entre telas distintas.

- **`hooks/`** — Hooks customizados que encapsulam lógica reutilizável: `useImageUpload` (manipulação de File API, validação e preview), `useDifficulty` (estado e derivações de dificuldade).

- **`routes/`** — Configuração das rotas da SPA via React Router DOM, definindo o mapeamento entre caminhos de URL e componentes de página.

- **`services/`** — Módulo responsável pela comunicação com recursos externos (APIs, File System API do browser). Isola as demais camadas de detalhes de implementação de I/O.

- **`utils/`** — Funções utilitárias puras (sem efeitos colaterais): validadores de tipo e tamanho de arquivo, formatadores de dados.

- **`types/`** — Definições TypeScript globais: interfaces, tipos e enums compartilhados entre camadas.

## 7. Padrões e Estilos Arquiteturais

Esta seção evidencia os **padrões e estilos arquiteturais** aplicados no projeto, com indicação do código que os comprova, conforme exigido na entrega (código comprobatório rodando).

### 7.1 Estilo Arquitetural: Componentes e Conectores (C&C)

O sistema segue o estilo **Component-and-Connector**, onde:

- **Componentes** são os arquivos `.tsx` (LevelSelection, ImageUpload, etc.)
- **Conectores** são as interfaces de comunicação: `props`, `Context API` e `hooks`

Este estilo é amplamente recomendado para aplicações React e favorece reuso e testabilidade independente.

### 7.2 Padrão Arquitetural: Camadas (Layered Architecture)

A organização `pages/ → components/ → hooks/ → services/` implementa uma **arquitetura em camadas**, onde:
- Camadas superiores dependem de camadas inferiores
- Camadas inferiores nunca dependem de camadas superiores (princípio da dependência unidirecional)

```typescript
// Exemplo em LevelSelection.tsx: camada de página usa hook (camada inferior)
const { preview, processFile, resetUpload } = useImageUpload();
//  ↑ páginas (camada alta) consomem hooks (camada mais baixa)
```

### 7.3 Padrão de Design: Context + Reducer (State Management)

O gerenciamento de estado global segue o padrão **Context + Hooks**, substituindo dependências externas como Redux para estado de escopo de aplicação limitado:

```typescript
// GameContext.tsx — padrão Context Provider
export const GameContext = createContext<GameContextType | null>(null);

export const GameProvider: React.FC<{ children: React.ReactNode }> = ({ children }) => {
  const [selectedLevel, setSelectedLevel] = useState<CardData | null>(null);
  const [difficulty, setDifficulty] = useState<DifficultyLevel>("medio");
  
  return (
    <GameContext.Provider value={{ selectedLevel, setSelectedLevel, difficulty, setDifficulty }}>
      {children}
    </GameContext.Provider>
  );
};
```

### 7.4 Padrão de Design: Custom Hooks (Separação de Lógica)

A lógica de negócio é extraída dos componentes visuais para **hooks customizados**, seguindo o princípio de separação de responsabilidades:

```typescript
// useImageUpload.ts — encapsula toda a lógica de upload
export const useImageUpload = () => {
  const [preview, setPreview] = useState<FilePreview | null>(null);
  const [dragging, setDragging] = useState(false);

  const processFile = useCallback((file: File | null | undefined) => {
    if (!file) return;
    if (!ACCEPTED_TYPES.includes(file.type)) return;
    if (file.size / 1024 / 1024 > MAX_SIZE_MB) return;
    // leitura e atualização de estado...
  }, []);

  return { preview, dragging, processFile, resetUpload, handleDrop, handleDragOver, handleDragLeave };
};
```

### 7.5 Padrão: CSS Variables + Design Tokens

O sistema de estilização utiliza **CSS Custom Properties (variáveis)** como design tokens globais, garantindo consistência visual e manutenibilidade do tema:

```css
/* Definição global (compartilhada entre todos os componentes) */
:root {
  --bg: #0f0f14;
  --surface: #1a1a24;

  --accent2: #f0a500;
  --success: #34c98a;
  --font: 'Outfit', sans-serif;
  --mono: 'Space Mono', monospace;
}
```
